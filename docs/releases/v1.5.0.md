# Heddle v1.5.0

Release date: 2026-05-16

Heddle v1.5.0 continues the foundation cleanup after the conversation-engine
work in v1.4.0. This release reorganizes the lower core layers that host
agent runs, approvals, runtime services, workspaces, daemon ownership, and
heartbeat scheduling.

The main theme is making service boundaries consistent across the core: class
entry points, local `types.ts` contracts, repository/schema-owned persistence,
and fewer one-off wrapper functions.

## Highlights

### Agent runs now have a clearer service boundary

The old large agent loop file has been split into focused classes for run
context, model turns, tool turns, finishing, memory checkpoint tracking,
mutation tracking, planning state, and small utilities.

Why this matters:

- the main agent run entry point is easier to inspect
- tool/model/finish behavior has clearer ownership
- future runtime changes can avoid editing one oversized execution file

### Approvals now follow the core service pattern

Approval policy handling, pending approval shape, remembered rules, codecs,
schemas, repository I/O, and service logic now live under a more explicit
approval domain structure.

Why this matters:

- persisted approval rules are validated at the repository boundary
- policy-chain behavior is easier to test and extend
- TUI and runtime callers have a cleaner core approval surface

### Runtime, daemon, credentials, and workspace services are separated

Runtime internals now distinguish the generic loop, tool registry, credential
resolution, daemon registry/host ownership, and workspace catalog behavior.
Heartbeat scheduling moved out of `core/runtime` into its own domain.

Why this matters:

- runtime is closer to a lower-level execution layer
- heartbeat no longer lives as a runtime subfeature
- daemon/workspace ownership paths have dedicated services and schemas

### Heartbeat now has a first-class domain

Heartbeat wake cycles, checkpoint-backed one-off runs, durable tasks, scheduler
loops, task state projection, run records, and host-facing views now live under
`src/core/heartbeat/`.

The current persisted task format is intentionally the clean refactored shape.
Older local heartbeat task files are not migrated; recreate old tasks with
`heddle heartbeat task add` if needed.

Why this matters:

- scheduled/background work has a clear home outside generic runtime code
- task state stores the latest wake result as one `state.result` object
- finished scheduler events include the durable run record instead of copying
  decision, outcome, summary, and usage into another flat payload

### Architecture docs now describe the intended layers

The chat layering document moved under `docs/architecture/`, and a new core
layering guide explains the desired direction for runtime, agent, chat engine,
heartbeat, memory, server, and web surfaces.

Why this matters:

- contributors have a concise map for where new behavior belongs
- future refactors have a shared target shape
- docs now match the service-boundary pattern used in the code

### Browser integration now covers heartbeat rendering

The browser integration fixture now includes a heartbeat task and saved run
record, and the Playwright suite verifies the Tasks screen renders that task,
run history, summary, and usage.
