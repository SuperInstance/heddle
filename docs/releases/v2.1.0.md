# Heddle v2.1.0

Release date: 2026-05-28

Heddle v2.1.0 advances the API-backed terminal rewrite and hardens the shared
control-plane run path used by browser and terminal clients. The release keeps
the existing production chat command in place while making `chat-v2` a more
complete proving ground for live session state, approvals, cancellation, and
prompt responsiveness.

## Highlights

### CLI v2 now runs on the control-plane API path

`chat-v2` now has an Ink-based terminal surface backed by shared tRPC client
code, live session event streaming, session refresh after run completion,
terminal activity updates, prompt draft stability, and run controls.

Why this matters:

- the terminal rewrite consumes the same session APIs as web-v2 instead of
  reaching into old TUI internals
- terminal users can see live activity, approvals, run state, and cancellation
  behavior through the control-plane path
- the old `chat` command remains the production fallback while `chat-v2`
  continues toward parity behind explicit commands

### Interactive prompt submission is now asynchronous

The control plane now accepts interactive prompts through an async session run
flow. Browser and terminal clients get an accepted run response quickly, then
sync durable messages, assistant streaming, tool activity, failures, and final
session state through live events plus run-state fallback polling.

Why this matters:

- interactive clients no longer need a long-running mutation to represent both
  "prompt accepted" and "agent finished"
- accepted user prompts are persisted server-side, so cross-client views stay
  aligned
- failure and cancellation paths can clear pending state predictably instead of
  leaving stale submitting or running indicators

### Shared client boundaries are clearer

Shared API-consumer code now lives under `src/client-shared`, with tRPC-derived
types, shared link/proxy construction, session activity shaping, approval
display metadata, and session-message behavior reused by web-v2 and cli-v2.

Why this matters:

- frontends share API contracts without importing core services or server
  controllers directly
- web-v2 and cli-v2 can stay aligned on run state and message semantics while
  keeping rendering code separate
- future clients have a clearer place to consume control-plane APIs

### Session activity and OpenAI streaming paths are more reliable

This release fixes stale daemon session leases, session activity status
clearing, browser submit status races, CLI v2 failed-run visibility, and OpenAI
completed-stream parser fallback behavior.

Why this matters:

- active sessions recover more consistently after run completion or failure
- browser and terminal clients are less likely to show stale activity or blocked
  run controls
- OpenAI OAuth-backed runs handle completed stream responses more defensively
